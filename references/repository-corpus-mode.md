# Repository and Corpus Mode

只在任务跨仓库、压缩包、多份文档、快照、冲突来源或当前状态结论时读取本文件。它补充主流程的第 2 至第 6 步，不重复通用语义解压规则。

目标是防止一篇流畅的说明把“写过、实现过、验证过、被接受、已交付”压成同一件事。

## R1. 限定材料边界并建立来源地图

默认只读。不要为了理解材料而执行未知脚本、安装依赖或修改工作树。只有当前状态结论确实需要验证，且用户授权、环境安全时才运行检查。

先记录：

- 仓库或素材包根目录；
- 快照、Commit、日期、dirty 状态、Overlay 或生成信息；
- 主要文件类型、目录和入口索引；
- 项目说明、治理规则、Agent 指南和文档索引；
- 与任务有关的 Authority、ADR、Spec、Ticket、代码、测试、Report、Claim、Handoff 和产品材料。

文件名和目录名只是导航线索，不自动证明内容当前有效。

**完成标准：** 能说明材料来自哪个边界、处于什么快照状态，以及哪些来源类型可能拥有本次问题的不同部分。

## R2. 写出 Coverage Contract

大材料不能用“我读过仓库”概括。开始深读前写下：

```text
材料总览：已知文件、目录、类型和规模
承重问题：这次输出必须回答什么
必读集合：直接拥有语义、决定、状态或证据的来源
全局搜索：需要跨仓库追踪的术语、状态和反证
抽样集合：按层级、时间或组件抽取的代表来源
未覆盖范围：没有读取、无法运行或无法验证的部分
```

按任务选择覆盖方式：

- 小型材料可以穷尽阅读；
- 端到端解释优先路径追踪；
- 术语漂移优先术语追踪；
- 大仓库采用分层抽样；
- 当前状态结论必须做反证搜索，主动寻找 `draft`、`candidate`、`not implemented`、失败检查和更弱的 Claim Ceiling。

**完成标准：** 实际读取、搜索、抽样和未覆盖范围都可陈述，不制造全量审计的假象。

## R3. 按问题域发现 Authority

先找项目自己声明的权威分工，不替项目发明一条通用文件优先级。对每个承重问题分别问：

- 谁定义正式术语和长期不变量；
- 谁拥有已采纳且难以逆转的决定；
- 什么说明当前实现；
- 什么记录实际观察到的行为；
- 什么拥有工作范围和生命周期状态；
- 什么只描述方向、假设或未来目标。

常见来源可以提供线索，但不能互相替代：

| 来源 | 通常能回答 | 通常不能单独证明 |
| --- | --- | --- |
| 语义根、SSoT、标准 | 定义、不变量、规范边界 | 当前代码已经实现 |
| ADR | 已接受取舍及理由 | 每日进度或运行结果 |
| 代码与配置 | 当前实现形态 | 规范已经改变或行为已验证 |
| 测试与执行证据 | 特定环境下观察到什么 | 未覆盖环境与产品价值 |
| Spec、Ticket、Tracker | 范围、承诺和工作状态 | Claim 已被接受或产品已交付 |
| Report | 当时实际执行和观察的内容 | 当前快照仍然通过 |
| Product、Roadmap、Research | 方向、目标和假设 | 当前能力事实 |
| Handoff、Chat | 增量线索和下一入口 | 长期 Authority |

来源冲突时保留冲突、各自问题域和依据。最新、最易读或最靠近根目录的文件不自动获胜。

**完成标准：** 每个承重问题都有本地 Authority 或明确的 Authority 缺口，没有用一个来源覆盖所有真相。

## R4. 维护 Source Ledger

为每个承重结论记录一行：

```text
Claim：准备在成品中表达的结论
Question Domain：它回答哪个问题
Statement Status：事实、决定、提案、假设、推断、未知或冲突
Authority Source：拥有该问题的来源
Evidence Source：实际观察或验证来源
Version / Freshness：Commit、日期、快照和 dirty 状态
Claim Ceiling：证据最多支持到哪里
Conflict：相反来源或未解决差异
Final Wording：符合证据等级的最终措辞
```

同一文件可以在一个问题上有 Authority，在另一个问题上只有线索。来源自己的推断与 Agent 的推断分开记录。

**完成标准：** 成品中的每个承重结论都能回到 Ledger，且措辞不超过 Authority 与 Evidence 共同允许的上限。

## R5. 通过 Current-State Gate

当前状态至少分成五层：

```text
工件存在
实现存在
当前验证已观察
结论已接受
能力已交付
```

这些状态不会自动晋升：

- 文件存在不等于工作流完成；
- 代码存在不等于能力经过验证；
- 静态检查通过不等于运行时行为成立；
- Candidate Claim 不等于 Accepted Claim；
- 历史报告通过不等于当前快照仍然通过；
- 产品方向明确不等于产品已经交付。

记录验证所处层级，例如静态、单元、局部运行、集成、真实 Provider、数据库重启、浏览器或生产。每项证据只支持它实际覆盖的环境、输入和时间点。无法运行的检查写成“未执行”，不要改写成“应当通过”。

**完成标准：** 每个“已有、可用、通过、完成、支持、生产就绪”类结论都有当前证据、环境和 Claim Ceiling；缺失部分保持未知。

## R6. 建立 Term Registry

跨文件术语发生别名、同名异义或层级漂移时，记录：

```text
Canonical Term
Definition Owner
Local Meaning
Alias
Deprecated or Confusing Term
Scope or Layer
```

统一表达时保留架构边界。项目级总称、正式对象名、代码类型名和产品名称不因相似就合并。第一次解释后，成品继续使用 Canonical Term。

**完成标准：** 每个承重术语都有定义拥有者和适用层级，别名不会被误写成同一对象。

## R7. 生成渐进披露的成品

仓库接手说明默认分四层：

1. **一屏全景：** 项目为何存在、主要层级、谁拥有哪类事实。
2. **一条真实路线：** 选一个请求、对象或决策，从入口走到结果与失败恢复。
3. **当前事实与目标状态：** 明确已经存在、已验证、已接受、仍是 Scaffold 或未来方向的内容。
4. **权威来源与继续工作：** 承重来源、未决冲突、覆盖声明、下一入口和需要补的验证。

README 只承担定位、全景、最短开始方式和深层链接，不复制整个知识库。承重结论附近保留 Source Pointer，避免把来源全部堆到文末。

**完成标准：** 新接手者能找到 Authority、走通一条真实路线、判断当前能力边界，并知道未覆盖与下一步。

## 仓库分支最终门禁

交付前逐项确认：

- Authority 是从项目中发现的，而不是由 Skill 预设；
- Coverage Contract 与实际工作一致；
- Source Ledger 覆盖所有承重结论；
- 当前状态没有跨越“存在、实现、验证、接受、交付”的层级；
- 历史来源带有时间和快照边界；
- Canonical Term、别名和层级没有被抹平；
- 结论附近能找到来源，未知和冲突保持可见；
- 用户提供的私有素材没有进入公开 Fixture 或示例。
